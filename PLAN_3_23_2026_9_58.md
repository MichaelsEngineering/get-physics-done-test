# Repository Plan: Sentinel Multiscale Field Dynamics

## 1. REPO NORTH STAR
A deterministic research-and-systems workspace for discovering common structure across multiscale physics, learned dynamics, and intervention-aware control, using executable abstractions that make invariants, stability, and rollout behavior first-class artifacts rather than after-the-fact analysis.

## 2. ARCHITECTURAL THESIS
The repository should embody the chain `geometry -> symmetry -> operator -> closure -> integrator -> controller -> evaluator` as both a scientific hypothesis and a software decomposition.

- `geometry`: what the state is and what can be conserved; artifact = typed state schemas, grid/graph/equivariant layouts, dataset manifests.
- `symmetry`: what transformations must be respected; artifact = resolver rules, equivariance/periodicity/locality contracts, validation checks.
- `operator`: resolved dynamics backbone; artifact = deterministic PDE kernels, stencil/spectral/message-passing operators, native kernel bindings.
- `closure`: learned or analytical correction for unresolved scales; artifact = trainable models, feature extraction, offline training/eval harnesses.
- `integrator`: rollout mechanism where stability and structure preservation are enforced; artifact = stepping APIs, invariant projection hooks, long-horizon diagnostics.
- `controller`: intervention policy over the rollout; artifact = regret-minimizing or constraint-enforcing decision modules that act on the same state interface.
- `evaluator`: common judge across physics, ML, and control; artifact = invariant dashboards, regime-shift suites, benchmark reports, trace readers.

Scientific mapping:
- Physics asks whether a common state/operator/controller formalism spans turbulence, plasma, and MHD.
- AI/ML asks whether closures or world models generalize across scales while respecting structure.
- Control asks whether interventions can improve stability or regret without violating physical constraints.

Engineering mapping:
- Each link in the chain becomes a stable package boundary with typed interfaces, reproducible configs, benchmarks, and traces.
- The repo stays honest by treating only grid turbulence as “validated core” until later phases explicitly promote new paths.

## 4. LANGUAGE AND SUBSYSTEM SPLIT
- Python: orchestration, typed configs, research iteration, CLI, evaluators, training loops, dataset manifests, trace readers, paper-ready reporting. This keeps iteration fast and the research surface legible.
- C: native periodic-grid kernels, pthread-based parallel stencil execution, shared-memory ring buffer, binary trace writer, Unix-domain-socket control plane. This is the strongest low-level signal and fits the simulation/runtime story naturally.
- C++: narrow, optional layer for RAII wrappers, typed trace-reader SDK, and native benchmark harnesses where object-oriented design is a real benefit. Keep it small and intentional; do not move core scientific logic into C++.
- Shell/Make: deterministic entrypoints for sync, build, format, test, benchmark, native demos, and report packaging. Use Make as the stable public interface even if CMake and `uv` sit underneath.

Default split:
- Research truth lives in Python and Markdown.
- Performance-critical kernels and transport live in C.
- C++ exists only where lifecycle management and typed composition clearly improve readability.

## 5. PHASED IMPLEMENTATION ROADMAP

### Phase 0: repo cleanup and invariants
- Objective: make the repo trustworthy before adding complexity.
- Deliverables: normalized top-level layout, public `research/` docs, config schema/versioning policy, run manifest format, test taxonomy, artifact and trace conventions, renamed CLI surface (`sentinel ...`) with backward-compatible alias.
- Acceptance criteria: every run has a config hash and seed; CI runs format/lint/type/unit/CLI smoke; docs clearly distinguish validated core vs scaffolds.
- Risks: over-reorganizing too early; breaking current GPD workflow.
- Do not build yet: new physics backends, training pipelines, native runtime.

### Phase 1: validated 2D turbulence core
- Objective: make grid turbulence the credible baseline.
- Deliverables: explicit `FieldState`, `Operator`, `Integrator`, `Evaluator` interfaces; deterministic periodic 2D rollout; invariant metrics; benchmark fixtures; dataset manifest for synthetic turbulence sequences.
- Acceptance criteria: repeated seeded runs are bitwise or tolerance-stable; invariant summaries are emitted every run; regression tests cover energy drift, mean preservation, and rollout sanity.
- Risks: interface churn if state semantics are still fuzzy.
- Do not build yet: graph parity, fancy controllers, plasma claims.

### Phase 2: invariant tracking and benchmark harness
- Objective: turn “works” into “measured.”
- Deliverables: benchmark suites for horizon length, energy drift, runtime, memory, and regime-shift slices; evaluator package; trace schema; run comparison reports; golden baseline registry.
- Acceptance criteria: `bench` command compares current run to baseline and fails on invariant or performance regressions outside configured thresholds.
- Risks: too many metrics too early; noisy baselines.
- Do not build yet: complex learned closures, distributed execution.

### Phase 3: learned closure experiments
- Objective: add ML where the interfaces can support honest comparison.
- Deliverables: closure training/eval API, baseline residual closure, one spectral-mixing closure, train/val/test split manifests, ablation configs, offline report generation.
- Acceptance criteria: closure experiments can be run from versioned config only; reports compare learned closure against no-closure and analytical baseline on long-horizon metrics, not just one-step loss.
- Risks: benchmark contamination, uncontrolled randomness, overclaiming generalization.
- Do not build yet: large model zoo, graph/equivariant productionization.

### Phase 4: control / regret layer
- Objective: introduce intervention as a first-class research dimension.
- Deliverables: `Controller` interface, simple constraint controller, simple regret-minimizing controller for bounded intervention, evaluator metrics for intervention cost and stability gain.
- Acceptance criteria: controller can act on the same rollout API; reports show no-intervention vs controller comparison; invariant violations are measured under intervention.
- Risks: confusing control with training; hiding instability with aggressive correction.
- Do not build yet: RL platform, MPC framework, multi-agent control.

### Phase 5: systems demonstration layer in C / multithreading / IPC
- Objective: add a native runtime that belongs to the science story.
- Deliverables: threaded C stencil kernel, shared-memory telemetry ring, Unix-domain-socket rollout monitor, Python bridge, native benchmark suite, binary trace format and reader.
- Acceptance criteria: native kernel can execute one benchmarked operator path and stream telemetry to a monitor process on Linux; integration tests verify protocol correctness and performance deltas versus Python baseline.
- Risks: systems work outgrows research priorities; portability distractions.
- Do not build yet: distributed cluster runtime, GPU-native stack, remote daemon orchestration.

### Phase 6: public portfolio polish
- Objective: make the repo impressive without becoming a demo dump.
- Deliverables: cleaned README, architecture diagrams, benchmark snapshots, example commands, one polished native demo, one polished research report, contributor guide.
- Acceptance criteria: external reader can understand north star, run a smoke path, inspect benchmarks, and see clear evidence of architecture plus low-level systems competence in under 15 minutes.
- Risks: style over substance.
- Do not build yet: blog-engine complexity, marketing-heavy artifacts.

## 6. RESEARCH TRACKS

### Physics track
- Core questions: which invariants must be exact vs monitored; what state variables survive extension from 2D turbulence to MHD; which rollout pathologies signal bad physics fastest.
- Minimal experiments: seeded 2D periodic turbulence rollouts, invariant-drift sweeps over timestep and viscosity, simple extension sketch with additional conserved quantities in config.
- Failure signals: drift grows with horizon in nominal regime; invariants depend on implementation path; extensions require ad hoc interface breaks.
- Progress evidence: stable invariant envelopes, consistent diagnostics across refactors, extension path documented without changing the core chain.

### AI/ML track
- Core questions: what closure target is most stable over long horizons; how much structure must be encoded vs learned; which regime-shift splits expose brittle models.
- Minimal experiments: no-closure baseline, residual closure baseline, one spectral/operator-style closure, train/test split by resolution or forcing regime.
- Failure signals: one-step loss improves while long-horizon drift worsens; closure only works on narrow seeds; learned model bypasses operator/integrator structure.
- Progress evidence: closure improves long-horizon metrics on held-out regimes without violating invariant envelopes.

### Control / regret track
- Core questions: can interventions reduce instability without masking physics; what regret definition is meaningful for rollout correction; when should controller act vs abstain.
- Minimal experiments: bounded correction controller, projection-enforcing controller, regret measured against invariant/stability objective over horizon.
- Failure signals: controller only “wins” by injecting energy or over-damping dynamics; intervention cost dominates benefit; policy is inseparable from closure internals.
- Progress evidence: measurable stability gain with bounded intervention cost and transparent invariant accounting.

## 7. SYSTEMS / LOW-LEVEL DEMO TRACK
Best subsystem: a native periodic-grid runtime consisting of a threaded C stencil operator plus shared-memory telemetry streaming and a Unix-domain-socket monitor.

Why this is the best fit:
- It belongs directly to the existing 2D turbulence vertical slice.
- It demonstrates C, pthreads, Linux IPC, binary formats, and performance measurement without adding an unrelated “systems toy.”
- It supports research: the same traces used for hiring-signal demos also support benchmark reproducibility and rollout introspection.
- It can naturally expose OOP design through a small C++ trace-reader/monitor SDK while keeping the computational kernel in C.

Subsystem scope:
- C kernel computes one operator step over periodic grid tiles using a deterministic thread partitioner.
- Telemetry writer emits step metrics and sampled fields to a shared-memory ring buffer.
- A monitor process subscribes via Unix domain socket control messages and drains shared-memory records.
- Python orchestrates runs, validates outputs, and compares native vs Python traces.

## 8. RECOMMENDED INTERFACES
```python
class FieldState(TypedDict):
    kind: Literal["grid", "graph", "equivariant"]
    channels: dict[str, Array]
    geometry_id: str
    time_index: int
    metadata: dict[str, str]

class Operator(Protocol):
    name: str
    def tendencies(self, state: FieldState, ctx: StepContext) -> FieldState: ...

class Closure(Protocol):
    name: str
    def correction(self, history: list[FieldState], ctx: StepContext) -> FieldState: ...

class Integrator(Protocol):
    name: str
    def step(
        self,
        state: FieldState,
        tendencies: FieldState,
        correction: FieldState | None,
        ctx: StepContext,
    ) -> FieldState: ...

class Controller(Protocol):
    name: str
    def act(self, state: FieldState, metrics: "InvariantReport", ctx: StepContext) -> "ControlAction": ...

class Evaluator(Protocol):
    name: str
    def measure(self, rollout: list[FieldState], ctx: RunContext) -> "EvaluationReport": ...

class NativeKernelBridge(Protocol):
    def available(self) -> bool: ...
    def step_many(self, init: FieldState, cfg: NativeKernelConfig, steps: int) -> "NativeRunResult": ...

class TraceSink(Protocol):
    def open(self, manifest: RunManifest) -> None: ...
    def emit_step(self, state: FieldState, metrics: dict[str, float], step: int) -> None: ...
    def close(self) -> "TraceIndex": ...
```

Supporting types:
- `StepContext`: seed, dt, step index, config hash, device/runtime info.
- `ControlAction`: additive correction, projection request, or abstain.
- `InvariantReport`: energy, enstrophy, divergence, mass, controller cost, regret terms.
- `EvaluationReport`: metrics, thresholds, pass/fail verdict, links to traces and artifacts.

## 9. REPRODUCIBILITY AND VALIDATION
Repo rules:
- Deterministic seeds: every run requires explicit seed; default seed is forbidden for benchmark/regression commands.
- Config versioning: every config has `schema_version`; incompatible changes create new versioned configs, not silent reinterpretation.
- Artifact naming: `runs/<date>/<problem>/<gitsha>_<confighash>_<seed>/`.
- Benchmark runs: always emit run manifest, environment snapshot, timing summary, invariant summary, and baseline comparison.
- Invariant tracking: energy, mean, and problem-specific invariants are logged at every rollout step; missing invariant output is a failed run.
- Trace capture: benchmark and native runs must emit binary or JSONL trace plus summary index.
- Test categories:
  - smoke: CLI and one-step end-to-end sanity
  - regression: seeded outputs and metric envelopes
  - physics-validation: invariant and conservation behavior
  - integration: Python/native bridge and telemetry protocols
  - benchmark-validation: threshold checks on runtime and memory
- Promotion rule: no new backend is called “supported” until it has smoke, regression, and physics-validation coverage.

## 11. HIRING-SIGNAL STRATEGY
The repo should signal competence through one coherent narrative: “I build research platforms where the scientific abstractions, systems runtime, and validation discipline line up.”

Visible signals:
- Low-level systems: native C kernel, POSIX shared memory, Unix sockets, deterministic threading, native tests.
- Concurrency: explicit worker partitioning, thread-safe telemetry, benchmark reports showing scaling behavior.
- Software architecture: clean interface chain from state to evaluator, narrow boundaries, versioned configs, compatibility rules.
- Scientific computing: invariant tracking, horizon-based evaluation, honest statements about validated scope.
- AI/ML research engineering: closure experiments wired into the same benchmark/evaluation path as physics baselines.
- Linux tooling: Make/CMake/uv integration, CI matrix, perf-oriented docs, reproducible run manifests.

Anti-pattern to avoid:
- separate “demo” folders unrelated to the science problem;
- many unfinished backends;
- notebooks as the only source of truth.
